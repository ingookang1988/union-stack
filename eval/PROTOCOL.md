# union-stack 효능 A/B 프로토콜

`scripts/health.js`(자기 준수도)와 `scripts/eval.js`(구조 레버리지 프록시)는 **하네스가 가치를 *가능케 하는가*** 를
측정한다. 그러나 그것이 **에이전트가 실제로 더 잘하게 만드는가**(효능)는 모델을 돌려야만 안다. 2026년 하네스
엔지니어링의 표준 명제 — *"모델이 아니라 하네스를 측정하라"*(SWE-bench·terminal-bench 계열) — 를 이 레포에
적용하는, 재현 가능한 최소 프로토콜이다. 채워진 레퍼런스 인스턴스 하나만 있으면 누구나 돌릴 수 있다.

## 0. 설계: 단일 변인 A/B
- **A (harness-on):** `.union-stack/` 평면 + `AGENTS.md` + 런타임 훅(`scripts/HOOKS.md`) 전부 활성.
- **B (harness-off):** 동일 코드베이스·동일 태스크, 단 평면을 숨기고 `AGENTS.md`를 빈 1줄 stub으로 치환(훅 off).
- 같은 모델·같은 태스크·같은 시드. 차이는 *하네스 유무* 하나뿐. 각 태스크 N=5회 반복(분산 측정).

## 1. 태스크 스위트 (레퍼런스 인스턴스에서 정의 — 값은 example)
프로젝트별로 채우되, 세 *함정 유형*을 반드시 포함한다(각 하네스가 막겠다고 주장한 실패에 1:1 대응):

| # | 태스크(example) | 노리는 실패 | 측정 지표 |
|---|---|---|---|
| T1 | "`WO-01a1-2` 인증 토큰 만료 처리 추가" | 같은 계보 과거 함정 반복(`LSN-01a`) | **재발 여부**(과거 실패를 또 냈나) |
| T2 | "공유 응답 타입에 필드 추가" | 기존 계약(`CON-01`) 무시·재정의 | **재사용률**(기존 자산 사용 vs 신규 생성) |
| T3 | "프로바이더를 X로 교체" | 폐기된 방향(`HISTORY` 기재)으로 회귀 | **회귀율**(버린 길로 다시 갔나) |
| T4 | 대형 plan 진입(깊은 계보) | 컨텍스트 비대→누락·드리프트 | **tokens-to-first-edit**, 과제 정확도 |

## 2. 측정 지표 (3대 가치에 정합)
1. **재사용률** = (기존 계약/픽스처 재사용 횟수) / (재사용+신규생성). `eval.js`의 reuse 신호가 *상한*을 주고, A/B가 *실현율*을 준다.
2. **회귀율** = 폐기 방향(`HISTORY`/거부된 `PRO-*`)으로 재진입한 태스크 비율.
3. **tokens-to-task** = 첫 정답 편집까지 소비 토큰(+ 총 토큰). 하네스의 컨텍스트 경제가 *순이득*인지(주입 비용 < 절감) 검증.
4. **정확도/리워크** = 인간 리뷰 통과까지의 수정 왕복 수.

> 보조 자동지표: 각 실행 후 `node scripts/eval.js --json`과 `node scripts/context-budget.js --json`을 스냅샷해
> 구조 프록시와 실측 효능의 상관을 추적한다(프록시가 효능을 예측하면 프록시만으로 회귀검사 가능).

## 3. 판정
- 성공 기준(example): A가 B 대비 재사용률 ≥ +30%p, 회귀율 ≤ B의 1/2, tokens-to-task 비악화(±10% 이내)면
  "하네스가 *조향*에서 순이득"으로 본다. 토큰이 크게 악화되면 §1 context-budget 상한을 조인다.
- 결과는 `archive_ledger.md`(전술) 또는 `project/HISTORY.md`(전략적이면)에 사실+근거로 기록한다.

## 4. 정직한 한계
- 모델·태스크 의존이라 절대수치는 이식 불가 — *자기 레포 내 A/B 델타*만 유효.
- 평가자(인간 리뷰) 편향을 줄이려 블라인드(어느 쪽이 A인지 모르게) 채점 권장.
- 이 프로토콜은 *설계*다. 실행 결과(실데이터)는 사적 자산(§자산 민감도)이므로 공개 레포에 싣지 않는다.
