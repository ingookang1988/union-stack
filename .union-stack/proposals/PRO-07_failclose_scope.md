# [PRO-07] Fail-close 스코프 한정

> 근거: E5 해악 팔 측정(`eval/RESULTS.md` 5회차, 2026-07-16). 요구 지식이 전부 국소인 양성 과제에서
> harness-on 팔이 Fail-close를 과적용해 **델타 −0.8**(on 1/5 vs off 5/5)을 기록했다. 원인은 규율
> 지식의 과일반화가 아니라(H2·H3 델타 0) **Fail-close 문언의 해석 여지 × 요청 전거 부재**의 상호작용.

## 1. 제안 요지
AGENTS.md 규칙 1(Fail-close)에 스코프 한정 문장을 추가한다(예시 문안):

> Fail-close는 요청이 규범·계약·HISTORY와 **충돌**하거나 요청 *내용*이 진짜로 모호할 때 발동한다.
> **인간의 직접 요청 자체는 "모호함"이 아니다** — 계보 문서에 전거가 없을 뿐인 양성 요청은 멈추지
> 말고 수행하되, 필요하면 한 줄 확인 질문 또는 스코프 노트를 답에 곁들인다.

## 2. 무엇이 바뀌나 (승인 시)
- `AGENTS.md` 규칙 1에 위 한정 문장 추가 (Schema — 인간이 반영, `Approved-by:` 트레일러).
- `scripts/HOOKS.md`의 훅 주입 문구 가이드에 동일 원칙 반영(E3 훅 메시지가 이 함정을 재생산하지 않도록).

## 3. 무엇이 바뀌지 않나
- 충돌 기반 Fail-close(zeta 회귀, Schema 편집, blast-radius 잠금)는 그대로 — E5의 H2·H3가 이 경계는
  이미 정확히 작동함을 보였다.
- 신규 도메인·기둥·게이트 없음(문언 수정만). Split-principle check 불요(구조 변경 아님).

## 4. 반증 가능한 검증 계획
- 승인 후 E5 H1 유형(약한 전거 + 국소 과제)을 수정 문안으로 N=5 재측정 → 델타 −0.8 → ≥ −0.2로
  회복되면 성공. 동시에 T3(회귀 방지) 재측정으로 충돌 기반 Fail-close가 무뎌지지 않았는지 확인
  (on 5/5 유지가 조건 — 한정이 과하면 이쪽이 무너진다).
