# Project History — 전략적 분기점 원장

> 과거 조회로 현재 project·roadmap을 이해하고, 과거에 폐기한 방향으로의 회귀를 방지한다.
> 각 항목은 **사실 + 근거**가 한 쌍이다. 근거가 회귀 판단의 핵심이다.

| 날짜 | 분기점(사실) | 근거(왜) | 시사점(고려사항) |
|---|---|---|---|
| (예시) 2026-01 | (예시) 의존성 A 도입 후 폐기 | (예시) 성능이 아니라 **라이선스 변경**으로 상용 사용 불가 | 라이선스가 풀리면 재검토 가능 — 영구 금지 아님 |
| (예시) 2026-02 | (예시) 모놀리식 → 모듈러 피봇 | (예시) 팀 확장으로 병렬 개발 필요 | 단일 개발자 회귀 시 재평가 가능 |
| 2026-06-08 | `topology/`→`architecture/`, `mechanism/`→`verification/` 폴더 개명 (v5.12) | 폴더명이 역할과 충돌("mechanism"이 아키텍처 용어와 혼동) — 채택자 피드백 | ID·계보는 불변, 참조만 이동. 명명 기준 확립: 폴더명 = 수행 역할 |
| 2026-06-15 | v6.0 승격 — 효능을 주장에서 측정으로 전환 (ADR-03) | 통제 A/B에서 "효능 ∝ 비국소성, 모델 강도 무관" 법칙 확인 | E3(enforce 실전 도그푸딩) 미완 채 승격 — 근거 확립 ≠ 운영 경화, 후속 필수 |

> ⚠️ 위 항목은 "하지 마라"가 아니라 "과거에 이랬으니 고려하라"다. 최종 판단은 현재 맥락에서.
