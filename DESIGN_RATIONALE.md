> **v4 → v5 개정의 본질**
>
> v4까지는 "7개 기둥(7-doc-stack)을 *어떻게* 운영하는가"의 명세였다. v5는 한 단계 아래로 내려가, **"왜 이 기둥들이며, 이것으로 충분한가(빈 구간이 없는가)"** 라는 질문에 답한다. 즉 v5의 기여는 새 기능이 아니라 **구조의 자기 정당화와 누락 식별**이다.
>
> 핵심 변화:
> 1. **분할 원리의 명시화** — 무엇을 독립 기둥으로, 무엇을 하위 문서로 가를지를 기준으로 정식화했다.
> 2. **기둥의 3층위 재분류** — 7개를 평평하게 보던 것에서, *액자(경계) / 칸(기둥) / 화살표(검증)* 라는 서로 작동 방식이 다른 세 층위로 재정의했다.
> 3. **두 개의 빈 구간 식별 및 충전** — 격자 분석으로 (A) 시간축 학습 평면(제품 오답노트), (B) 공유 계약·재사용 자산 평면(정적 명세 + 테스트 인프라)이 비어 있었음을 발견하고 채웠다.
> 4. **제4 권한 계층(Proposal) 도입** — 하네스가 자기 자신을 진화시키는 메타 루프를 위해, 기존 3계층(Read-only / Atomic-Write / Append-only)에 *제안(Proposal)* 계층을 추가했다.
>
> **v5.0 → v5.1 추가분:**
> 5. **분할 기준에 5번째 축(자산 민감도) 추가** — 기술적 분류와 직교하는 "공개 가능 / 소유 필요" 축을 정식 편입했다(§1.6).
> 6. **테스트 인프라의 3분할** — 테스트를 단일 자산이 아니라 *전략(규범)·도구(계약)·케이스(실제)* 세 층으로 분해해 각기 다른 기둥에 배치했다(§5 재작성).

---

## 0. 이 문서의 독해 순서

v5는 결론(기둥 목록)부터 제시하지 않는다. 결론은 원리에서 *연역*되어야 누락을 검사할 수 있기 때문이다. 따라서 다음 순서로 읽는다.

1. **분할 원리(§1)** — 무엇을 기둥으로 가르는가의 기준.
2. **3층위 모델(§2)** — 기둥/경계/검증의 구분.
3. **격자와 빈 구간(§3)** — 원리로 격자를 깔고, 빈칸을 식별.
4. **충전된 두 평면(§4, §5)** — 빈칸을 채우는 새 구성요소.
5. **제4 권한 계층(§6)** — 자기 진화 루프.
6. **통합 디렉터리·미결 사항(§7, §8)**.

## 1. 분할 원리: 무엇이 독립 기둥의 자격을 갖는가

v4까지 "왜 기둥이 6개였다가 7개인가"는 직관에 의존했다. v5는 이를 4개의 직교 기준으로 정식화한다. 어떤 지식 덩어리를 만났을 때, 이 기준들을 적용하면 *독립 기둥인지 / 기존 기둥의 하위 문서인지 / 아예 격자 밖 메타 요소인지* 가 결정된다.

### 1.1 기준 1 — 변화 속도 (Lifecycle Velocity)

지식이 얼마나 자주 변하는가. 거의 불변(규범·계약), 자주 변함(상태·작업), 추가만 됨(증거·기록)으로 갈린다. **변화 속도가 크게 다르면 같은 문서에 두지 않는다.** 자주 변하는 것이 거의 안 변하는 것을 오염시키고, 에이전트가 둘을 함께 읽는 비용이 낭비되기 때문이다.

> 이 기준이 plan을 독립시킨 근본 이유다. 전술적 기획 의도(자주 생멸)가 불변 규범과 한 문서에 섞여 컨텍스트를 오염시켰기에 분리했다. 또한 이 기준은 3계층 권한 모델(Schema/Wiki/Raw)의 분류 축 그 자체이기도 하다.

### 1.2 기준 2 — 상태 대 행위 (State vs. Action)

지식이 "무엇이 존재하는가(명사·결과)"를 다루는가, "무엇을 하고 있는가/되어가는가(동사·과정)"를 다루는가. feature(무엇이 살아있다)와 sprint(무엇을 하는 중)는 변화 속도가 비슷하지만 이 축에서 갈린다.

### 1.3 기준 3 — 추상화 레벨 (Abstraction Level)

지식이 "이래야 한다(당위·규범)"인가, "이렇다(실제·관찰)"인가. topology(당위)와 그것이 구현된 현실(실제)은 같은 대상을 다루지만 추상 레벨이 다르다. 부모-자식 위계(topology → feature, roadmap → plan/sprint)가 이 축에서 나온다.

### 1.4 기준 4 — 계약 (Contract)

당위와 실제 사이에 끼인 **제3항**. "이래야 한다(규범)"도 "이렇다(현실)"도 아닌, **"이렇기로 *약속했다*"**. 공유 타입·인터페이스·enum, 테스트 인프라의 사용 규약이 여기 속한다. 계약은 규범에서 연역되지 않는다 — 누군가 *정하면* 진실이 된다. 그리고 일단 정해지면 코드 전체가 물리적으로 의존하므로(시그니처 하나 바뀌면 컴파일이 깨짐), 규범보다 더 강하게 박혀야 한다. **규범 위반은 "나쁜 코드"를 낳지만, 계약 위반은 "안 도는 코드"를 낳는다.**

### 1.5 적용 우선순위

새 지식을 분류할 때 기준은 다음 우선순위로 적용한다.

1. **먼저 종류(대상)로 가른다** — 제품에 관한 것인가, 하네스(운영) 자체에 관한 것인가. 종류가 다르면 물리적으로(폴더로) 분리.
2. **그 안에서 변화 속도로 파일 분할 여부를 정한다** — 자주 변하면 하위 격리 파일로 쪼개고, 드물게 변하면 한 파일에 두고 *용법(섹션)*으로만 구분. (미리 쪼개지 않는다 — YAGNI)

> 이 우선순위는 "테스트 인프라"와 "오답노트" 논의에서 실제로 도출되었다. 둘 다 "에이전트가 기존 자산을 못 찾는" 증상이 같았으나, 1차 기준(대상: 제품 vs 운영)으로 가르니 서로 다른 평면에 속함이 드러났다.

### 1.6 기준 5 — 자산 민감도 (Asset Sensitivity) *[v5.1 신설]*

위 네 기준은 모두 *인지적·기술적* 축이다. 다섯 번째 기준은 그것들과 **직교하는 전략적 축**이다: 이 지식이 **공개해도 되는 방법론인가, 소유해야 하는 경쟁 자산(moat)인가.**

이 축이 필요한 이유는 오픈소스화 검토에서 드러났다. 코드를 공개하면 복제 가능해져 코드 자체는 더 이상 해자가 아니지만, **테스트 케이스 인프라·축적된 교훈은 복제되지 않는다.** 그것은 "이 코드가 무엇을 만족해야 하는가"에 대한 누적 지식이며, 포크한 경쟁자는 코드를 가져가도 그 안에 응결된 엣지 케이스·회귀·도메인 함정은 못 가져간다. 따라서 같은 코드를 가져도 *변경의 속도와 안전성*에서 격차가 벌어진다. 오픈소스 시대에는 산출물(코드)이 아니라 *능력*(안전하게 진화시키는 힘)이 자산이다.

같은 대상이 기술적으로는 "계약 층"이면서 전략적으로는 "핵심 소유 자산"일 수 있다. 그러므로 이 축은 분류를 *덮어쓰지 않고 겹쳐서* 적용된다. 세 등급으로 나눈다.

1. **순수 공개 (Pure Template):** 메커니즘·규율·문법·정규식 등 *방법론*. 내용을 한 톨도 담지 않으며, 예시조차 가공 더미(`PLAN-01a_example`)로만 둔다.
2. **순수 사적 (Pure Instance):** 실제 도메인 지식 — 테스트 케이스, 실제 ID 계보, 오답노트 내용, 데이터 리니지. 절대 비공개. 진짜 해자.
3. **경계 위험 지대 (Leakage Zone):** 틀처럼 보이나 내용이 스며들기 쉬운 것 — 카탈로그의 "예시", 디렉터리 구조에 박힌 도메인명, 오답노트의 카테고리 분포. 대부분의 누설은 1·2가 아니라 *3을 부주의하게 다룰 때* 일어난다. 공개 시 더미로 치환·정화한다.

> 이 축은 §2의 project(정체성 경계)과 종류가 다른 *자산 경계*이며, 공개·소유 분리를 판단하는 기준이 된다.

## 2. 3층위 모델: 기둥은 평평하지 않다

v4는 7개를 동일한 `.md` 문서로 취급했다. 그러나 분할 원리로 재검토하면, 그중 일부는 *지식을 담는 칸*이 아니라 성격이 근본적으로 다른 요소임이 드러난다. 세 층위로 재분류한다.

### 2.1 액자 (Boundary) — project

`project`(`[프로젝트명].md` 형태의 기술 백서)은 격자의 한 칸이 아니다. **격자 전체가 *무엇에 관한* 것인지를 정하는 경계**다. "이 프로젝트는 무엇이고 무엇이 아닌가"를 정의한다.

에이전트에게 두 역할을 한다. (1) **경계 판별기** — 들어온 의도가 이 프로젝트의 스코프 안인지 거른다. (2) **어휘 사전** — 이 프로젝트만의 도메인 용어를 정의해 에이전트가 정확히 알아듣게 한다. 주로 세션 부트스트랩 시 1회 주입되며, 매 작업마다 참조되지는 않는다.

> 백서의 두께는 개발 철학과 연동된다. 린스타트업(MVP만 정의)이면 얇고 경계가 흐릿하지만, 선기획·대량개발(풀스코프를 미리 그림) 방식에서는 두꺼워지고 그 안에 경계와 어휘가 모두 담긴다.

### 2.2 칸 (Pillars) — 지식을 담는 기둥들

실제로 지식을 저장하는 기둥. v5에서 §3의 격자가 이들을 좌표 위에 배치한다. 제품 차원과 운영 차원으로 나뉜다.

### 2.3 화살표 (Verification) — mechanism

`mechanism`은 칸이 아니다. **칸과 칸 사이의 정합성을 끊임없이 재는 동적 검증 평면**, 즉 칸을 잇는 *화살표*다. 그리고 단일 화살표가 아니라 *여러 검증의 묶음*이다.

- **규범↔현실 검증 (Drift):** topology(당위)와 실제 코드가 일치하는가 → `gap`이 담당.
- **계약↔현실 검증 (Contract):** 약속한 타입/시그니처대로인가 → FLOW의 self-correction이 담당.
- **약속↔결과 검증 (Promotion):** 하기로 한 것이 실제로 되었는가 → evidence가 target을 live로 승격.

**mechanism의 정의(v5):** 관찰된 현실(입력)을 *외부에서 받아*, 칸들 사이의 정합성을 검증하고(처리), 어긋남만 기록하는(출력) 순수 검증 엔진. **현실을 *생성*하거나 *저장*하지 않는다.** 진짜 Raw(테스트 로그·AST)는 CI/CD·컴파일러가 생성하는 외부 신호이며, mechanism은 그것을 잠깐 받아 잴 뿐이다.

> **[v4 정합]** 이는 v4의 "`.mechanism/raw/`(입력)와 `.mechanism/derived/`(출력) 분리"를 *왜* 분리해야 하는지 설명한다. 관찰(evidence·state)은 화살표의 *입력*이고, 검증 결과(gap)는 화살표의 *출력*이다. 입력과 출력을 한 폴더에 두면 "Raw인가 Derived인가"가 흐려진다. v5는 한 발 더 나아가, raw에 담기는 것조차 "외부에서 들어온 신호의 임시 수신부"이지 mechanism의 영구 소유물이 아님을 명확히 한다.

## 3. 격자와 빈 구간의 식별

분할 기준 중 가장 근본적인 두 축(추상화 레벨 × 상태/행위)으로 격자를 깔면, 칸들의 자리와 *빈칸*이 동시에 드러난다.

### 3.1 제품 차원 격자

|  | **상태 (명사 · 존재)** | **행위 (동사 · 변화)** |
|---|---|---|
| **당위 (불변, 이래야 한다)** | `topology` 아키텍처 규범 | `roadmap` 어디로 가야 한다 |
| **계약 (약속됨)** | **▣ 빈칸 → §5에서 충전** | `plan` 무엇을 하기로 했다 |
| **실제 (관찰됨, 이렇다)** | `feature` 무엇이 살아있다 | `sprint` 무엇을 하는 중 |

격자가 즉시 드러내는 것:

- **대칭의 빈칸.** "당위" 행과 "실제" 행은 상태·행위 두 칸이 모두 차 있다. 그러나 **"계약" 행은 행위(plan)만 있고 상태가 비어 있다.** plan이 "무엇을 *하기로* 약속했나(행위쪽 계약)"라면, 비어 있는 칸은 "무엇이 *이러하기로* 약속됐나(상태쪽 계약)" — 즉 공유 정적 명세와 재사용 자산이다. **이 빈칸은 plan을 떼어낸 것과 완벽히 대칭이며, 격자가 8번째 구성요소의 존재를 예측한 것이다.** → §5에서 충전.

- **격자에 안 들어가는 것.** mechanism과 project은 본문 칸을 채우지 못하고 빠진다. 이는 결함이 아니라 §2의 3층위(화살표/액자)임을 재확인한다.

### 3.2 시간축의 부재 — 두 번째 빈 구간

위 격자는 전부 *현재의 한 스냅샷*을 다룬다 — topology는 지금의 규범, feature는 지금 사는 기능, sprint는 지금 하는 일. 그러나 **"같은 실수의 반복"은 여러 스프린트에 걸쳐서만 보이는 패턴**이며, 어떤 단일 스냅샷에도 나타나지 않는다.

즉 기존 격자는 **공간축(구조)은 촘촘하나 시간축(반복·학습)이 비어 있다.** mechanism이 *공간축 검증*(지금 규범과 현실이 맞나)이라면, 그 시간축 짝 — *과거에 같은 실패를 겪었나* — 이 없다. → §4에서 충전.

> 이 평면의 개념적 씨앗은 1차 명세의 4단계 메모리(working/episodic/semantic/procedural)에 이미 있었다. "여러 세션에 걸쳐 반복 확인된 패턴이 semantic memory로 승격된다"고 적었으나, 그것을 담을 기둥을 만들지 않았다. v5는 이론에 저장소를 부여한다.

## 4. 충전 ① — 시간축 학습 평면 (제품 오답노트)

### 4.1 정체와 자리

mechanism이 *사후에 공간축을 검증*하는 화살표라면, 이 평면은 ***사전에 시간축을 경고*** 하는 짝이다. 같은 계보에서 반복된 실패(디버그 케이스, 도메인별 함정)를 누적하고, 미래의 같은 작업에 *진입하기 전에* 힌트로 선주입한다. 제품에 묶이므로 ZFS ID로 색인되며, 격자 안(제품 차원)에 속한다.

### 4.2 주입 시점 — Upward Fetching의 시간축 확장

오답노트는 *사후*가 아니라 *사전*에 읽혀야 가치가 있다(시험 전에 봐야 하는 오답노트). 따라서 독립적 검색이 아니라 **Upward Fetching에 묶인다.** 에이전트가 `WO-01a1-2`를 받아 부모(`01a1`, `01a`, `01`)를 역산해 PLAN·FLOW를 끌어올 때, 한 줄을 추가한다: *"이 계보에서 과거에 반복된 실패가 있는가?"* 그리하여 코드 작성 전에 "공간(부모 맥락) + 시간(과거 실패)"을 함께 주입한다.

### 4.3 고유 관리 규율 — 가장 위험한 기둥

이 평면은 다른 기둥과 달리 **검증되지 않은 정보가 추론의 맨 앞(프롬프트 선주입)에 꽂히며, 대조할 현실이 아직 없다.** 잘못된 힌트 하나가 뒤따르는 모든 토큰 해석을 왜곡한다. 따라서 고유한 규율이 필수다. 네 규율 모두 *기존 시스템 원리의 재적용*이다.

1. **진입 장벽 (반복 임계):** 1회 실패는 교훈이 아니다. 최소 2~3회 반복 관찰된 것만 등재한다. 1회짜리는 그 스프린트의 gap으로 닫고 끝낸다. *(메모리 승격 원리)*
2. **만료 (유효 근거 결속):** 각 힌트는 "왜 아직 유효한가"의 근거와 묶인다. 근거가 사라지면(예: 해당 함정을 topology가 아키텍처 레벨에서 강제하게 바뀜) 자동 은퇴 후보가 된다. *(forgetting curve 원리)*
3. **스코프 (계보 한정):** 힌트는 자기 ZFS 계보에만 매달린다. `01a1`의 힌트가 `02` 작업에 딸려오면 길을 잃게 만든다. §4.4의 검증된 자식/형제 판정으로 무관 계보의 힌트를 원천 차단한다. *(ZFS 스코프 원리)*
4. **비용 상한 (극단 압축):** 선주입되므로 길면 그 자체가 낭비다. 서사가 아니라 경고다. "토큰 만료 처리 누락 주의 (3회) → [EVD-01a1-3] 참조" 한 줄. 상세는 ID 링크로 위임. *(결정화 원리)*

### 4.4 생명주기 관리 — 북키핑 부활의 함정

규율 2(만료)의 판단 주체가 문제다. 에이전트가 자동 은퇴시키면 유효한 힌트를 잘못 죽일 위험(과소 경고), 인간이 일일이 관리하면 없애려던 북키핑 노동의 부활. **해법은 §6의 제4 권한 계층과 연결된다.** 에이전트가 "이 힌트는 무효인 듯합니다(topology가 바뀌었으니)"를 *제안*하고 인간이 승인하면 은퇴 — mechanism이 drift를 다루는 방식 그대로. 자동 은퇴의 위험과 수작업 북키핑을 동시에 피한다.

## 5. 충전 ② — 공유 계약·재사용 자산 평면

### 5.1 정체 — §3.1이 예측한 "상태쪽 계약"

격자에서 비어 있던 칸(계약 × 상태)이다. 두 내용물이 같은 성격으로 이 칸에 들어간다.

- **공유 정적 명세:** 시스템 전역이 의존하는 타입·인터페이스·enum·핵심 시그니처. (당초 `ssot`로 채우려던 것)
- **공유 테스트 인프라:** 테스트 러너, 공통 fixture, mock 서버 등 *영속적 검증 도구*.

세 기준으로 자리를 확인하면 — 변화 속도(거의 불변 ✓ 계약층), 상태/행위(상태 ✓), 종류(계약: "이렇게 쓰기로 약속됐다" ✓) — 모두 "상태쪽 계약"을 가리킨다. **이 평면의 본질적 성격은 "에이전트가 새로 만들면 안 되고, 찾아서 따라야 하는 것들"이다.** 존재 이유는 *재사용 강제*다.

### 5.2 테스트의 3분할 *[v5.1 재정의]*

"테스트 인프라"는 단일 자산이 아니라 **추상 레벨이 다른 세 층**이며, 각기 다른 기둥에 산다. 이를 한 덩어리로 취급하면 규범과 도구가 섞여 흐려진다.

| 층 | 내용 | 추상 레벨 | 거처 | 관리·변화 |
|---|---|---|---|---|
| **전략 (왜·어떻게)** | "도메인은 인프라 없이 단위 테스트한다. 외부 의존은 포트로 추상화·mock한다. E2E는 이 경계에서만." | 당위(규범) | **topology** | 인간이 정함, 거의 불변 |
| **도구 (무엇으로)** | 테스트 러너, 공통 fixture, mock 서버, 헬퍼의 *호출 규약* | 계약 | **본 평면(§5)** | 재사용 강제, 자동 추출 가능 |
| **케이스 (이 기능은 무엇을 통과해야)** | 특정 기능의 구체적 테스트 명세 | 실제 | **FLOW Contract 옆**(`.feature/flow/`, 같은 ZFS ID) | 기능별, TDD 순서로 코드 전 작성 |

이 3분할은 두 직감을 *동시에* 충족한다. "테스트는 아키텍처의 한 축이어야 한다"(전략 → topology, 클린 아키텍처의 의존성 규칙과 같은 층) **그리고** "테스트 자산은 좌표화되어 재사용돼야 한다"(도구 → 본 평면). 충돌이던 둘이 분업이 된다.

> **왜 전략이 topology인가:** 클린 아키텍처의 의존성 역전(DIP)·포트-어댑터 구조의 *존재 이유 자체*가 상당 부분 "도메인을 인프라 없이 테스트하기 위함"이다. 즉 테스트 가능성은 아키텍처의 부산물이 아니라 그 구조를 떠받치는 원인이다. 테스트 전략과 아키텍처는 같은 결정의 양면이므로 같은 층(topology)에 산다.

### 5.3 인프라가 케이스를 생성한다 — 도구가 케이스를 포섭

도구(층 2)와 케이스(층 3)의 관계가 핵심이다. **도구는 케이스를 *생성하는 규칙*이다.** 에이전트가 인프라의 관용구를 알면 개별 케이스는 그 틀을 채우는 행위로 자연히 수렴한다. 인프라를 모르면 케이스마다 제각각 새로 만들어 중복이 발생한다. 따라서 케이스는 별도 관리 대상이 아니라 도구의 *인스턴스*다 — 인스턴스를 일일이 관리하지 않고 *틀*을 관리한다.

### 5.4 카탈로그의 필수 3요소

도구 카탈로그에서 "존재한다"만으로는 재사용을 못 한다. 에이전트가 호출법을 몰라 여전히 새로 만들 수 있기 때문이다. 각 자산은 반드시 세 가지를 함께 명세한다: **무엇이 있고(존재) · 어디 있고(경로/ID) · 어떻게 쓰는가(호출 패턴·예시).** 세 번째가 빠지면 카탈로그는 작동하지 않는다.

### 5.5 참조 시점 — 가시성 문제, 강제가 아닌 좌표화 *[v5.1 해소]*

v5.0에서 미결이던 "언제·얼마나 참조를 강제할지"는 실사용 관찰로 해소되었다. **명시적으로 "있는 것을 쓰라"고 지시하면 재사용률이 사실상 100%이고, 지시 없으면 반반이다.** 즉 에이전트는 재사용을 *거부*하는 것이 아니라 자산의 *존재를 인지하지 못해서* 새로 만든다. 문제는 의지가 아니라 **가시성**이다.

따라서 하드 게이트(커밋 전 조회 증명 강제)는 과잉이다. 처방은 **좌표화 + 부트스트랩 가시화**다. 도구 카탈로그를 ZFS로 좌표화해 두면, 에이전트가 작업 시 시스템을 스캔할 때 자연히 시야에 들어온다. 별도의 강제 장치 없이 가시성만 확보하면 재사용은 따라온다.

### 5.6 카탈로그의 동기화 — 사전 분리로 누설 차단

카탈로그가 현실보다 낡으면 오히려 중복을 *유발*한다("카탈로그에 없네 → 새로 만들자"). 동기화 주체 문제의 해법은 두 가지다.

- **자동 추출 (권장):** 카탈로그를 손으로 쓰지 않고, 테스트 도구가 정해진 규약(지정 디렉터리·네이밍·docstring 태그)을 따르게 한 뒤 스크립트로 코드베이스에서 추출한다. 코드가 진실이고 카탈로그는 그 투영이므로 *추출이 곧 동기화*다. 비용은 "규약 준수"이며, 그 규약은 topology(전략 층) 또는 본 평면에 박는다. 이는 mechanism이 외부 신호(evidence)를 자동 수신하는 원리와 동일하다.
- **제안 기반 (대안):** 에이전트가 카탈로그에 없는 신규 도구를 감지해 등재를 *제안*하고 인간이 승인(§6 제4 권한 계층). 자동 추출이 어려운 비정형 자산에 적용.

> 자산 민감도(§1.6) 관점에서, 카탈로그 *구조*는 공개 가능하나 그 안의 *실제 도구 호출 예시*는 경계 위험 지대다. 공개 시 더미 도메인으로 정화한다.

## 6. 제4 권한 계층 — 하네스 자기 진화 루프 (Proposal)

### 6.1 두 개의 자기 갱신 루프

v4까지 "자기 갱신"되는 것은 *제품 지식*뿐이었다. 에이전트가 evidence를 읽고 feature를 갱신하는 식이다. 정작 **하네스의 규칙(Schema) 자체는 자기 갱신되지 않았다** — 인간만 손대는 Read-only 성역이었다. 여기에 두 번째 루프가 빠져 있었다.

- **루프 1 (기존, 빠름):** 현실 → 에이전트가 읽음 → *제품 위키*(feature, mechanism) 갱신. 매 이벤트마다. — *행동(behavior)*.
- **루프 2 (신규, 느림):** 반복된 실패·교훈 → *하네스 자신*(Schema·프로세스)을 진화. 여러 스프린트에 한 번. — *진화(evolution)*.

루프 1은 개체가 환경에 반응하는 것, 루프 2는 종이 세대를 거쳐 적응하는 것에 대응한다. "뭔가 비는데"라는 감각의 정체는 루프 2의 부재였다. Karpathy LLM Wiki의 정신("위키가 자기 자신을 갱신한다")을 제품을 넘어 하네스 자신에게까지 적용하는 것이다.

### 6.2 수위 선택: 제안(2번)

자기 진화 루프는 본질적으로 "에이전트가 Schema를 바꾸자고 말하는" 행위이며, 이는 "Schema는 Read-only·Fail-close"라는 신성한 원칙과 충돌한다. 세 수위 중:

1. 관찰만 — 안전하나 너무 느림.
2. **제안까지 — 에이전트가 초안을 올리고 인간이 승인/거부. ◀ 채택.**
3. 자동 진화 — 강력하나 Fail-close 원칙 위반.

**2번 채택의 근거는 이미 작동하는 패턴의 일반화라는 점이다.** 에이전트는 이미 *제품* drift에 대해 "임의로 고치지 않고 gap에 기록한 뒤 정합성 패치를 제안"하고 있다(실사용에서 빈번히 관찰됨). 하네스 회고는 그 동일 메커니즘을 *하네스 자신*에 적용하는 것뿐이다. 새 발명이 아니라 검증된 패턴의 확장이므로 가장 안전한 진화다.

### 6.3 제4 권한 계층의 정의

기존 3계층에 네 번째를 추가한다.

| 계층 | 대상 | 에이전트 권한 |
|---|---|---|
| Read-only | Schema | 읽기만 |
| Atomic-Write | Wiki | 행 단위 정밀 수정 |
| Append-only | Raw / Ledger | 추가만 |
| **Proposal (신규)** | **Schema·하네스 규칙에 대한 변경 제안** | **쓴다 — 단 승인 전 효력 0** |

작동: 에이전트가 제안을 *쓴다* → 승인 전에는 **효력이 없다** → 인간이 승인하면 Schema로 *승격*, 거부하면 **기각 사유와 함께 보존**된다.

### 6.4 거부 기록의 보존 — 무한 반복 차단

**거부된 제안을 삭제해서는 안 된다.** 에이전트는 같은 드리프트를 또 관찰하므로 같은 제안을 반복한다. 거부 사유가 없으면 무한 루프("게이트 추가하자 → 거부 → 한 달 뒤 또 제안")가 발생한다. 사유가 보존되면 에이전트가 먼저 읽고 "이건 이래서 기각됐다"를 인지해 되풀이를 멈춘다.

이는 시스템의 **Supersession 원칙**("과거 지식을 삭제하지 않고 대체 관계를 명시")을 *의사결정 과정 자체*에 적용한 것이다. 그리고 **이 제안-결정 로그가 곧 하네스 회고의 실체다.** 별도 "회고.md"를 만드는 것이 아니라, 제안→승인/거부→사유의 누적이 그대로 "하네스가 어떻게 진화해왔고 무엇을 시도·기각했나"의 감사 추적(audit trail)이 된다. 회고는 *문서*가 아니라 *결정의 누적*이다.

### 6.5 제품 학습(§4)과 하네스 학습(§6)의 분리

두 평면은 메커니즘이 같다(반복 관찰 → 누적 → 미래 참조). 그러나 **대상이 다르므로 물리적으로 분리한다**(분할 우선순위 1번: 종류 우선). 제품 학습은 격자 *안*(ZFS ID 색인, 빈번한 갱신), 하네스 학습은 격자 *밖*(메타, 드문 갱신)이다.

단, 하네스 학습은 갱신 빈도가 낮으므로 **파일을 쪼개지 않고 한 평면 안에서 용법(섹션)으로만 구분**한다(분할 우선순위 2번: 변화 속도). 빈번해지면 그때 쪼갠다(YAGNI).

### 6.6 세션 이어달리기 (Handoff) *[v5.1 추가]*

핸드오프는 세션 종료 시 다음 세션 에이전트에게 넘기는 이어달리기 바통이다. **시간축이되 주기가 다르다** — `.lessons/`가 여러 세션에 걸친 반복 패턴(느린 시간축)이라면, 핸드오프는 바로 다음 세션으로의 1회성 인계(빠른 시간축)다.

- **거처:** 새 기둥이 아니라 `.sprint/HANDOFF.md`. 핸드오프는 "현재 작업의 세션 경계 스냅샷"이며, sprint(실제·행위, Wiki)와 변화 속도·종류가 같으므로 합친다(분할 원리). 별도 디렉터리를 파면 "현재 작업 상태"가 두 곳으로 쪼개져 응집이 깨진다.
- **휘발성:** 최신 하나만 유효. 매 세션 덮어쓰고 직전 것은 `prev.md`로 롤링. 영속 교훈은 `.lessons/`, 결정은 archive_ledger로 따로 승격.
- **작성 주체:** 세션을 마치는 에이전트가 스스로 생성(Wiki 원자적 쓰기). 따라서 *생성 규율*이 템플릿에 박힌다.
- **형식 원칙:** 인간이 읽을 일기가 아니라 *다음 에이전트가 이어받을* 기계 친화 구조. 변경 위치는 산문이 아니라 **ZFS ID 목록**으로 적어, 다음 세션이 그 ID로 Upward Fetching을 돌려 맥락을 스스로 복원한다. 모든 것을 설명하지 않고 *어디를 보면 되는지*만 가리킨다(극단 압축, §4.3과 동형).
- **필수 5요소:** 세션 요약 · 변경 위치(ID 목록) · 다음 작업(단일 진입점) · 미해결/주의 · 검증 상태.
- **부트스트랩 순서:** 새 세션은 project(정체성) → HANDOFF(이어달리기) → ID 기반 Upward Fetching 순으로 진입한다.

> 핸드오프 작성 행위는 "이번 세션 회고"의 축소판이지만, 제품 작업의 연속성을 다루므로 메타 평면(.lessons/.proposals)과는 층이 다르다.

## 7. 통합 구성 요소 지도 (v5)

대화로 발굴한 전체 구조를 3층위로 정리한다.

```
[ 액자 / 경계 ]
  project ([프로젝트명].md)   프로젝트 정체성·경계·도메인 어휘 (기술 백서)
  README.md                    7-doc-stack + 신규 평면들의 사용법

[ 칸 / 기둥 — 제품 차원 격자 ]
                    상태(존재)            행위(변화)
  당위(불변)        topology              roadmap
  계약(약속)        ★공유계약·자산평면(§5)  plan
  실제(관찰)        feature               sprint (+HANDOFF.md 세션 이어달리기)
  시간축(반복)      ★시간축 학습평면(§4)    —     (mechanism의 시간축 짝)

[ 화살표 / 검증 — 동적 평면 ]
  mechanism  (외부 현실을 받아 칸 간 정합성을 검증, 어긋남만 출력)
    ├─ raw/      외부 신호 임시 수신 (CI/CD·컴파일러가 생성, 에이전트 read-only)
    └─ derived/  검증 출력: gap(규범↔현실), state(구조 관찰)

[ 메타 / 자기 진화 — 제4 권한 계층 ]
  Proposal 평면  하네스 규칙 변경 제안 → 인간 승인/거부 → 사유 보존 (= 하네스 회고)
```

★ = v5에서 새로 충전된 두 평면.

**[v5.1] 자산 민감도 오버레이 (직교 축):** 위 모든 요소는 (공개 가능한 *틀* / 소유할 *내용*)으로 다시 갈린다. 틀 = Schema 계층 ≈ 공개 템플릿, 내용 = Wiki·Raw 계층 ≈ 사적 인스턴스.

**[v5.1] 테스트 3분할 — 한 자산이 세 거처에 산다:**
```
  전략(왜·어떻게)  → topology       (당위, 인간, 거의 불변)
  도구(무엇으로)    → §5 계약평면      (계약, 자동추출, 재사용 강제)
  케이스(통과조건)  → FLOW 옆(§5.2)   (실제, 기능별 ZFS ID, TDD 순서)
```

> **명명에 관한 주석.** §0에서 밝힌 목표(범용화·용어 다듬기)에 따라, ★ 두 평면과 Proposal 계층의 *최종 명칭*은 실무 표준 용어로 다듬는 별도 작업을 권한다. 후보: 시간축 학습평면 → *Lessons / Pitfalls Ledger* 또는 *Troubleshooting KB*; 공유 계약·자산 평면 → *Shared Contracts & Fixtures* 또는 *Platform Catalog*; Proposal 계층 → *RFC / Change Proposal Log*. 본 명세는 구조를 확정하고 명칭은 열어둔다.

## 8. 미결 사항 (다음 버전에서 결정)

### 8.1 ~~공유 계약·자산 평면의 참조 시점·강도~~ → §5.5에서 해소됨 *[v5.1]*

실사용 관찰("지시하면 100% 재사용, 미지시 시 반반")로 *가시성 문제*임이 확인되어, 하드 게이트가 아닌 **좌표화 + 부트스트랩 가시화**로 결론. 상세는 §5.5 참조.

### 8.2 명칭 표준화

§7 주석의 신규 평면·계층 명칭을 실무 표준 용어로 확정.

### 8.3 격자의 완전성 재검증

§3 격자가 "당위/계약/실제 × 상태/행위"로 6칸을 정의했고 그중 빈칸을 채웠다. 차기 검토: 이 2×3이 *정말* 완전한가, 혹은 제3의 축(예: 보안·관측 같은 횡단 관심사)이 별도 차원으로 필요한가. 분할 원리(§1)를 적용해 "X 없는 시스템은 없다 + 기존 칸 어디에도 안 들어간다"를 만족하는 후보가 있는지 주기적으로 재검사.
